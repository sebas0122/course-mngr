# SQLModel Migration Summary

This document describes the migration from dataclasses and pandas DataFrames to SQLModel for the course manager application.

## Overview

The application has been successfully migrated to use SQLModel, a library that combines the simplicity of Pydantic with the power of SQLAlchemy. This provides better type safety, validation, and database integration.

## Changes Made

### 1. `course.py` - Course Model

**Before:** Used a dataclass with basic type annotations
**After:** SQLModel-based class with proper field definitions

Key changes:
- Inherits from `SQLModel` with `table=True`
- Uses `Field()` for column definitions with descriptions
- Maps to the `materias` table in the database
- Proper handling of `ARRAY(Integer)` for the `profesor` field (list of professor IDs)
- Added `get_codigo()` method to generate course codes
- Spanish column names maintained for database compatibility (`nombre`, `horario`, `aula`, etc.)

### 2. `professor.py` - Professor Model

**Before:** Used a dataclass with validation in `__post_init__`
**After:** SQLModel-based class with validation in `__init__`

Key changes:
- Inherits from `SQLModel` with `table=True`
- Maps to the `profesores` table
- Validation for `contratacion` (hiring type) and `formacion` (education level)
- Spanish column names maintained (`nombre`, `identificacion`, `correo`, etc.)
- Class-level constants for valid values remain unchanged

### 3. `courses_functions.py` - Database Operations

Updated all functions to work with SQLModel objects:

#### `retrieveDBTable(supabase, table_name)`
- Now returns a list of SQLModel objects (Course or Professor) instead of DataFrames
- Automatically converts database rows to the appropriate model
- Falls back to DataFrame for unknown tables (backwards compatibility)

#### `getClassesList(courses_list, semester)`
- Changed parameter from DataFrame to list of Course objects
- Iterates over Course objects instead of DataFrame rows
- Accesses properties directly from Course objects (e.g., `course.nombre`, `course.nivel`)
- Uses `course.get_codigo()` method for course codes

#### `getProfessorsData(supabase)`
- Returns a dictionary mapping professor IDs to their info
- Works with list of Professor objects instead of DataFrame
- Accesses properties directly from Professor objects

#### `addProfessorToDB(supabase, professor)`
- Accepts a Professor SQLModel object
- Uses `professor.model_dump()` to convert to dictionary for database insertion
- Excludes `None` id values when inserting new records

### 4. `main.py` - Main Application

Updated variable names and function calls:

- Changed `dataframe` to `courses_list` throughout
- Updated all calls to `getClassesList()` to use `courses_list` instead of `dataframe`
- Updated `open_add_class_window()` to work with Course objects instead of DataFrame rows
- All Course properties accessed using the SQLModel field names

### 5. `dnd.py` - Drag and Drop

No changes required - the dnd module works with the info dictionaries that are still generated by `getClassesList()`, maintaining compatibility.

## Benefits of SQLModel

1. **Type Safety**: Better IDE support and type checking
2. **Validation**: Automatic validation of data types and constraints
3. **Database Integration**: Direct mapping to database tables
4. **Pydantic Features**: JSON serialization, validation, and more
5. **Maintainability**: Clearer data models with explicit field definitions
6. **Documentation**: Field descriptions provide self-documenting code

## Database Schema Mapping

### Course (materias table)
- `id`: Primary key
- `nombre`: Course name
- `facultad`, `dependencia`, `materia`: Course code components
- `grupo`: Group number
- `tipo`: Course type (e.g., "T-P")
- `es_lab`: Boolean indicating if it's a lab
- `nivel`: Semester/level
- `horario`: Schedule string (e.g., "L16-18|M8-10")
- `profesor`: Array of professor IDs
- `aula`: Room/classroom
- Additional fields: `ide`, `horas_teoricas`, `horas_practicas`, `horas_tp`, `electiva`, `es_dept`

### Professor (profesores table)
- `id`: Primary key
- `nombre`: Professor name
- `identificacion`: ID number (unique)
- `correo`: Email address
- `catedra`: Boolean for cathedra status
- `contratacion`: Hiring type
- `formacion`: Education level

## Testing

To test the changes:

1. Ensure SQLModel is installed: `pip install sqlmodel`
2. Run the application: `python main.py`
3. Verify that:
   - Courses load correctly for each level
   - Professor information displays properly
   - Adding/editing/deleting courses works
   - Database operations complete successfully

## Notes

- All Spanish column names have been preserved for database compatibility
- The application maintains backwards compatibility where possible
- No changes to the Supabase database schema are required
- The info dictionaries used by the UI remain unchanged for compatibility
